## 在场段流程（严格顺序）

### 步骤 1 · 逻辑日期判定

**逻辑日期口径（权威源，weekly-review / close-node / dream 指针至此）**：物理 hour < 06:00 → 逻辑日期 = 物理日期 − 1；≥ 06:00 → 同物理日期。

只用于总结性归属——`_本周.md` 流水日期、daily / weekly / close-node 的日期字段与周次判定、排梦任务包 prompt 正文。**cron 排程字段不适用**（cron 按物理钟触发，排程时刻一律填物理时间）。

### 步骤 2 · 项目结论 / 阶段决策 → 工作库

扫描本次对话：结论确立 / 悬置点明确 / 阶段决策 / 框架更新 / 跨项目影响指针。如有 → 生成确认表（信号 / 内容摘要 / 建议写入位置）呈现。[用户称呼]在场回复 → 按确认写入项目权威源；无响应 → C 级结论不写入、挂起、继续。写入路由：`_progress/` → 调用 `write-progress`；新建 `.md` → 调用 `new-file`。

### 步骤 3 · 更新 长期记忆.md 当前处境

主要矛盾或项目状态有实质变化 → 覆盖更新 `长期记忆.md §当前处境`；无变化 → 跳过。

### 步骤 4 · 节点闭合兜底扫描

本次对话存在已事实闭合但未执行 close-node 的节点 → 按 `close-node` skill 补跑（无人值守例外口径见其 §触发）；无 → 跳过。

### 步骤 5 · 写入 _本周.md 今日进展

在 `_本周.md §进展记录` 追加今日工作段：

```markdown
### 周X YYYY-MM-DD

**项目名称**（`项目根目录路径`）
关联：`具体文件1.md` · `具体文件2.md`

- 今天做了什么 / 确立了什么决策 / 推进到哪里。
```

流水只记动作和决策，不复写理论推导细节。这一步同时是 dream 回放深度与有效工作日时钟的数据源。

### 步骤 6 · 排梦

定稿今晚 dream 任务包并排定：

1. **载荷判断**：逻辑日期为周日 → 任务包带「周日载荷」标记（→ `weekly-review/SKILL.md §周日梦载荷`）。
2. **关机判断**：默认带关机步；[用户称呼]本轮说过「留机 / 别关机」→ 不带。
- 排梦前检查当前会话模型：低于 Opus 4.8 档（如 Sonnet）→ 先提醒[用户称呼]切换模型再排梦（dream 继承排梦时刻的会话模型；无人值守环节智能下限不降档）
3. **排定**：一次性 **durable** cron（落盘持久化），触发时刻 = 当前物理时间 +3 分钟左右（趁缓存温、道别后会话即空闲）。**cron 排程字段一律填物理时间**——凌晨跨日时严禁把逻辑日期填进日期字段，逻辑日期只写进 prompt 正文。prompt 写明：执行 `dream` skill 完整流程 + 载荷标记 + 关机指令 + 逻辑日期。
4. **排后核验（硬步骤，禁止跳过）**：排定后立即尝试回读 cron 任务列表，核对下次触发时刻在未来且距当前 ≤10 分钟；不符 → 删除重排。
   - **CronList 调用失败（malformed / 工具解析错误）→ 降级核验**：不阻断流程，改用替代证据链——① CronCreate 返回了有效 job ID ② dream 实际触发后 last_dream 探针 / MEMORY_LOG 条目作事后地面证据；步骤 7 道别时明示「CronList 核验降级，以 CronCreate 成功回执为替代证据，明早 week-sync 昨梦核对接住」。
   - **返回若标 session-only（durable 未生效）**：照排不撤，步骤 7 必须明示「今晚的梦可能挂不住（durable 失效），明早 week-sync 昨梦核对接住补扫」——不静默。核验通过（或降级完成）前不得进入回应道别。
5. 架构 / 协议变更类动作（若当日发生且未记）→ 主 agent 定稿 ITERATION_LOG 条目派storage-agent即时落盘，不入梦（→ `ITERATION_LOG.md §ITERATION_LOG 收录边界`）。

### 步骤 7 · 回应道别

在场段全部完成后，自然语言回应[用户称呼]的告别语，附一句今晚梦的安排（跑什么载荷、是否关机）。
