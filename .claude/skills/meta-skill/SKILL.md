# meta-skill

把判断工作流蒸馏成 SKILL.md。按第 1→9 步顺序执行，不得跳步。

## 输入契约

- 蒸馏**已到手的素材**：前置访谈记录、正反案例集、现役 skill 文本、或已成型的判据库。素材里"哪些算成功案例 / 哪些算失败案例"的标注，视为人已锁定的给定，不在本流程内重判。
- 不现场访谈、不向用户追问。素材不足时走第 5 步推理；仍不足则标红交回素材端补料，不自行编造判据。
- 输出只有两种合法形态：一份过第 8 步的 SKILL.md，或一份「指明哪个判断节点要回去补料」的诊断。

## 总则 · 产出纯命令式（贯穿全程，第 8 步强制复查）

写进 SKILL.md 的每一句，只对执行 agent 下令"做什么 / 怎么判 / 满足什么走哪",不向它叙述"我们在做什么 / 这个 skill 是什么 / 为什么搞这套"。

- 删一切元层叙事：项目背景、设计意图、"本 skill 目的是…"、"我们的思路是…"。执行端只读到命令。
- 需要依据才能让 agent 在未覆盖情境里判对时，把依据熔进命令的条件或后果，不另起描述句。写"按 1→2→3 执行不得换序：换序触发 X"，不写"保序很重要因为这步脆弱"。
- 本铁律对 meta-skill 自身同样生效：本文件即以命令式书写。

## 总则 · 编译者不自测（第 8 步控制流的元约束）

- 第 1-6 步（编译）由执行本流程的主 agent 做。第 8 步机械闸 + 白箱审主 agent 自做（读结构，非自证）。
- 第 8 步黑箱测主 agent **不得自跑**：派独立 sub-agent 盲跑、收报告、做裁断。理由熔进命令——编译者对自己的产出有盲区、顺素材滑、证伪不了自己；独立盲跑是编译者供不出的证伪源。

## 第 1 步 · 锁定目标函数与选择环境

先写死这份 skill 优化什么。缺目标函数，后面判据全退化成无法收敛的描述。填满下表，缺项不得进第 2 步：

```text
最终产物：     这份 skill 跑完交出什么
成功条件：     什么算做对
失败条件：     什么算做砸
选择环境：     谁用结果 / 谁判结果 / 结果进什么场景
可接受复现率：  100% 规则复现 / 还是企业场景复现 80-90% 关键工作即够
当前证据类型：  显式规则 / 历史案例 / 成功失败差异 / 专家陈述 / 相关性线索 / 反例（逐项标置信度）
```

允许用相关性与 likelihood 立判据，不得把相关性写成已证因果。不追求科学因果，只把命中率提到目标函数够用。

> 第 1 步产物是蒸馏脚手架，不写进成品 SKILL.md（写进去 = 向 agent 叙述"我们优化什么"，违反命令式总则）。

## 第 2 步 · 切执行段与判断节点

把整条工作流切成交替序列。逐项标类，不得留未标项：

- **执行段** = 步骤够清就能复现的操作，跑完留下可观测的阶段成果。
- **判断节点** = 拿前面成果对照目标做判断，判断结果直接改变后续执行走向。

按此式排出全流程，每项一句话标类（招投标示例）：
```
1. [执行段]   读招标文件 → 生成整合信息
2. [判断节点] 判该不该投 → 投进 3 / 不投出否决报告
3. [执行段]   按模板生成投标文件
4. [判断节点] 判中标率是否过线 → ...
```

把切分结果标为 inferred：这一刀切错（判断节点误当执行段、漏节点）会让下游全部判据白挖且不自暴露。交第 8 步黑箱测专项核这一刀。

## 第 3 步 · 判断节点展开四相位，定位并挖出三件套

把每个判断节点展开成一个四相位小循环：

- **1A 检索**：取新信息 X。取什么由节点目标定，执行性，不挖判据。
- **1B 整合**：把 X 并入已有信息 R，得 X×R。
- **2A 生成**：基于 X×R 生成结构 C，把信念 B1 更新为 B2。
- **2B 审核**：用 B2 量 C，定过线与否，给下一段执行指令。

按相位定位三件套，逐节点挖齐，缺一即判该节点未拆到位、回本步重挖：

| 件 | 产出相位 | 挖什么 |
|---|---|---|
| **判断标准** | **1B + 2A** | 尺子本身——凭什么这么判，落到一条可复述规则，不许"看情况" |
| **触发钩子** | **2B** | 阈值 / 分界线，就地给一个具体数或具体线，不许"差不多" |
| **路由** | **2B** | 每个分支各去哪，逐分支写明去向 |

判断标准只产于 1B-2A（尺子成形），钩子与路由只产于 2B（拿尺子量）；2B 不自产标准，B2 由 2A 给。挖三件套按此分相取料，不在四相位平均用力。

每个判断节点出一张卡：
```
判断节点：该不该投这个标
  判断标准（1B+2A）：预估利润率÷投入工时是否过线 + 有无控标迹象
  触发钩子（2B）：    利润率 < 8% 或 识别到控标
  路由（2B）：        过线 → 进应答表生成；不过 → 输出"放弃+理由"
```

## 第 4 步 · 判据来源，分流 index-mode / extract-mode

对第 3 步每条判断标准，先判它有没有现成权威源的家，再决定第 6 步怎么落：

- **判据已有权威源的家**（如记忆系统的规则文件、已成型的判据库）→ **index-mode**：本步只**定位**到判据所在的具体节，不在 skill 里重挖、不复制本体。第 6 步写精确指针。
- **判据无家、须从素材现挖** → **extract-mode**：本步按下方差集 + 六类来源挖出判据本体。第 6 步内联（因为没有别的家）。

误把 index-mode 判据当 extract-mode 挖出来内联 = 制造复制体，违反目标系统单一权威源，第 8 步白箱审 A1 会判 fail。先分流再挖。

**extract-mode 挖判据——判断标准来自差集，不来自单边共性：**

一条特征要成判据，必须**成功案例普遍有、失败案例普遍无**。成功共性单看会混入伴随特征（中标标书都用蓝封面，但废标也用蓝封面 → 蓝封面出局）。标准间一致性查不出伪判据（伪判据内部可自洽），只有比对失败案例的差集能切掉。共性与差集是同体两面：真共性的定义里已含"失败无"，写进 skill 时显式写成差集运算，防实现时滑回"只扫成功找重复"。

按此出判准卡：
```text
判准：       （成功有∩失败无 的那条特征）
来源：       显式标准 / 目标函数反推 / 成功-失败差异 / 反例边界（标出处）
程度阈值：    不止有无二值——成功侧强到何值才算数，给线
证据：       支撑这条的正反案例
反例：       看似合标准却失败 / 看似不合却成功
适用边界：
置信度：
更新条件：
```

允许相关性判准，标注为相关性、不伪装因果。

## 第 5 步 · 缺料时的不完全信息推理

第 3-4 步挖不动时，不得停、不得编通用答案。按 loop 走，推理受素材约束：

1. 追问 what：这节点处理什么对象 / 事实 / 输入。
2. 追问 why：这事实为何影响判断，与目标函数什么关系。
3. 追问 how：让这判断重演一次，证据怎么走到结论。
4. 生成 2-4 个事实假说或判准假说。
5. 每个假说写预测：成立则素材应出现什么证据，不成立应出现什么反证。
6. 选当前解释力最高一条，不平均对待。
7. 派 2-3 个 sub-agent 独立判：只给原始素材和问题、不给主模型结论，各自判哪个假说最合素材。
8. 多路收敛 → 立为 inferred 判准；分歧明显 → 回第 4 步重拆。
9. 所有推出的判据标 inferred + 置信度 + 更新条件。

案例稀疏到连差集都抽不出的节点（罕见致命点、全新无先例）：不得用收敛假装拿到判据。标红"案例不足 / 判据 inferred / 低置信"，回流素材端补正反案例。补料前留成人类判断点，写清判断者需先看到什么。

## 第 6 步 · 编码成 SKILL.md

把第 2-3 步的结构落成文字，按总则纯命令式写：

- **index-mode 判据** → 写**精确指针**，指到权威源具体节，不抄本体。
- **extract-mode 判据** → 内联写全（无别的家）：执行段写确定动作（取什么值、什么格式、什么校验、失败怎么办）；判断节点三件套写全。
- **inferred 判据** → 标 inferred + 置信度 + 更新条件，不写成硬规则。
- **人类判断点** → 写"此处交人判断" + 列判断者所需信息，不写成假规则。

两条收尾：
- **自由度配脆弱度**：一步多解、靠上下文 → 给原则让执行者自走；一步易错、须固定顺序 → 写死命令。
- **依据熔进命令**：要讲依据时熔成命令的条件或后果，不堆描述、不堆无源禁令。"不要做 X"仅当执行者真会自发做 X 才留，否则删。

**成品**：一份 SKILL.md。frontmatter = name + description（"做什么 + 何时调我"，第三人称带主动；"何时用"全进 description，不进正文）。正文 ≤500 行，超了把细节挪 `references/` 一级直链、正文留指针。多个原子重复同一确定动作 → 抽成 `scripts/` 脚本供调用。

包结构：
```
skill-name/
├── SKILL.md       必须
├── scripts/       可选 · 确定性任务（执行，不读入上下文）
├── references/    可选 · 按需读（长清单 / 判准本体 / schema）
├── agents/        可选 · 第 8-9 步派的 sub-agent prompt（grader / comparator / analyzer）
├── eval-viewer/   可选 · 第 8-9 步人审 HTML 生成器
└── assets/        可选 · 输出用模板
```

## 第 7 步 · description 触发 eval

description 决定 Claude 何时调用本 skill。编码完必过本步两道——judge 文字筛 + 行为闸触发实测，两道全绿才算触发层合格、才进第 8 步。description 任何改动，回跑本步。

1. 造触发用例集 `trigger_set.json`（schema 见 `references/schemas.md §11`）：20 条起——8-10 条 should_trigger（多样措辞 + 隐含需求 + 该赢的竞争 skill 场景）+ 8-10 条 should_not_trigger，后者要 **near-miss**（共享关键词但实则要别的），别放显然无关的。琐碎一步任务不是好测例（Claude 只为难独力搞定的任务查 skill）。
2. 起 workspace 骨架：`python scripts/init_eval_workspace.py "<skill名>"` → 填 `trigger_eval/trigger_set.json`。
3. 人核用例标注（可选）：`python eval-viewer/generate_review.py --trigger "<ws>/trigger_eval" --skill-name <名> --static <out.html>`，浏览器核 near-miss 标对没。
4. **切分 + 出判定卡**：`python scripts/run_trigger_eval.py prepare --skill "<名>" --skill-path "<新skill文件夹>" --runs 3`。它 60/40 切 train/test（固定 seed 可复现）、读被测 description、写 `_judge_card.md`。**不调模型**。新 skill 不在 C 盘 → `--skill-path` 必传。
5. **派 sub-agent 判定**：主 agent 读 `_judge_card.md`，按卡**派 N 个独立 sub-agent**（默认 3，各 clean context、互不可见），每个判全部用例、把判定写回 `trigger_eval/judging/original/run{1..N}.json`。判定走 sub-agent、**不经 subprocess/CLI**（起 `claude` 子进程会撞未登录、换机不可复现，与第 5/8 步同一套路）。
6. **计分**：`python scripts/run_trigger_eval.py score --skill "<名>"`。多数票计分，写 `result.json`（各版本 train/test + best）。**不调模型**。
7. **行为闸（触发地面真值 · 架在 judge 之上，不替代 judge）**：judge 是单 skill 二元判定，只筛 description 文字有没有把该触发场景写全，触发力强弱它判不出。真实触发靠对象层实测——把目标 skill 连同语义相邻的若干 skill 的 description 一并列给 clean-context sub-agent，给一句用户话问"调哪个 / 不调"，不告诉它在测什么（"评估该不该触发"的元层问法会让模型从主题推意图、虚高放行）。取 should_trigger 与 near-miss 各若干句，关键句多轮看一致：落对目标 = 过，落错邻居 / 漏触发 = fail。**哨兵守卫**：测试集塞一条已知错域 / 缺该场景的 description，断言行为闸必判它 fail；判过 = 行为闸自己失灵（竞争集没摆够 / judge 噪声），先修测试再信读数。**产出落盘**：判定写 `trigger_eval/behavior_gate.json`——每 query 记 `query / expected_skill / routed_to（各轮）/ consistent / pass`，与 judge 的 result.json 并列，可复现可审计，禁止只在对话里眼看收敛。
8. **不过的分流（judge 或行为闸 fail 才走 · 按失败类型选修法、非一律改写 · 至多 5 轮）**：先认失败类型再定修哪：
   - **漏触发**（should_trigger 没触发）→ description 没写出该场景 → 改本 skill description 回跑。
   - **抢错邻居 / near-miss 落错家** → 两 skill 边界问题 → 裁该收窄自己还是放宽邻居（**可能改的是邻居 description、不是本 skill**），判不清交人定归属，不默认改自己。
   - **哨兵判过** → 行为闸自己失灵（竞争集没摆够 / judge 噪声）→ 修测试集，**不碰 description**。
   迭代命令：定为改写则 `prepare --version rev1 --rev-desc "<改写>"` → 再派 sub-agent 判 + 重跑行为闸 → `score --version rev1`。**best_description 按 test 分择优**（防过拟合）。
9. 产出 `best_description`：**交人确认**后替换 frontmatter（脚本不自动改 skill）。

> judge 验文字层（场景写全没）、行为闸验触发层（竞争下路由对没），两道全绿才算触发层合格，与第 8 步执行层检验正交。description 改了回跑本步。

## 第 8 步 · 检验与回流

按 skill 解剖顺序过四道。前两道必跑（任何 skill 都有文件夹 + 内部逻辑），写全在此；后两道按 skill 特征条件触发，本体下沉 `references/verification.md`，此处只下命令式触发句。机械闸 + 内部逻辑审 + 指针审主 agent 自做（读结构，非自证）；黑箱测主 agent 派独立 sub-agent、自己只裁断（见总则·编译者不自测）。各道 grader 归属见 `references/verification.md §grader 三类`。

### 第一道 · 机械闸（必跑 · 跑脚本不靠肉眼）

跑 `python scripts/validate.py "<待审 skill 文件夹>"`：查 frontmatter 能否解析 / 必填字段齐否 / 路径有无死链 / 文件夹有无杂物 / description 是否纯触发句。退出码非 0 = fail，先修机械层再进下一道。

### 第二道 · 内部逻辑审（必跑 · 逐项 pass/fail · 交 LLM 判）

任何 skill 都有内部判断节点逻辑，本道必跑。逐项过三项：

**局部判准一致（逐判断节点）**：每个判断节点的判断标准必须展开成四环链，缺环 / 断链 = fail：
```
  目的     ：本节点服务第 1 步目标函数的哪一条（为达成什么而判）
→ 判断标准：基于该目的，凭什么判（一条可复述的尺子，禁"看情况"）
→ 判断指标：基于该标准，落到什么可测的数 / 线 / 信号（禁"差不多"）
→ 路由     ：该指标取值如何映射到各分支去向（逐分支写明）
```
自洽检验逐环验：目的真能生成该标准 / 标准真能量出该指标 / 指标真能驱动该路由——任一环接不上 = fail（标准与目的脱节 / 指标量不出标准 / 路由不依指标）。

**全局判准一致**：每个节点的"目的"都回溯得到第 1 步全局目标函数（无悬空节点）；节点间判准不互斥（A 节点判准不与 B 节点冲突）；无死结（改好甲必违乙 → 第 6 步迭代 loop 不收敛）；无对象挂起（每个被判对象都有来源 + 去向）。

**命令式复查**：grep 元层叙事词（"目的""我们在做""这是为了""本 skill"）→ 命中 = fail；逐句判命令句（令 agent 做什么 / 怎么判 / 走哪）vs 描述句（叙述是什么 / 为什么）→ 描述句 = fail；依据须熔进命令的条件 / 后果，不另起描述句。

### 第三道 · 指针审（条件：skill 有 references / scripts 才跑）

`skill 引用了外部权威源（references / scripts）→ 下沉 references/verification.md §指针审，跑复制体检测 / 指针有效性 / 被引片段自洽三项。`
`skill 无任何外部指针（纯 extract-mode 内联）→ 本道空过，跳过。`

### 第四道 · 黑箱测（真跑 · 主 agent 派 sub-agent 自己裁断）

**轻档 smoke（地板 · 必跑）**：派 3 个独立 sub-agent（各 clean context、互不可见、不给主 agent 结论），各跑**一个不同的真实任务**（3 路 = 3 个独立任务，覆盖不同判断节点 / 边界），skill 含不可逆动作 → 令其 DRY RUN（声明动作不真做）。各自产出逐判断节点路由结论 + 卡壳清单（(a) 顺指针读权威源仍判不动 / (b) skill 该说没说）。主 agent 据报告裁断：跑得通否 / 路由顺否 / 抓到哪些内逻辑审漏掉的执行 bug；必读 transcript 分清是 skill 错还是 grader 拒了合法解。

**重档收敛（条件：skill 高 stakes 才触发）**：
`skill 判错担责 / 要求接近 100% 复现率（接第 1 步 可接受复现率）→ 追加下沉 references/verification.md §黑箱测·重档（收敛测：同任务 × 3 路 + pass^k + 已有案例库切 held-out）。`
方法学（任务集构造 / 测试集来源降级链 / pass^k / 阈值 / DRY-RUN 特化 / 裁断追因）全在 verification，本步只决定该不该触发它。

**任一道不过 → 回第 1-3 步重拆，不得原地改字**。把诊断当新信息：哪个判断节点依据没挖透 → 回去补料 / 追问 → 重拆 → 再编码。

**最终产物**：过适用各道的 skill，或一份指明"哪个判断节点回去补料"的诊断。

## 第 9 步 · Improve 模式（迭代回流，已上线 skill 加固时走）

第 8 步判出"不过"或人审给了改进反馈后，按此迭代——不靠肉眼"觉得改好了"，靠盲评 + 胜负账定。

1. **改 skill**：从分叉追因 / analyzer 建议落具体改动。原则：从反馈泛化（别过拟合单个用例）、prompt 保持精简、讲"为什么"而非堆 ALWAYS/NEVER。
2. **重跑进新一轮**：`python scripts/init_eval_workspace.py "<名>" --iteration <N+1>` 摊目录，全任务（含基线）重跑。新 skill 基线 = without_skill；加固已上线 skill 则基线指向上一版或现役版。
3. **盲评对比**：派 comparator sub-agent（`agents/comparator.md`）双盲比新旧两版产物，定 winner/loser/tie，写 `comparison.json`。
4. **揭盲追因**：派 analyzer sub-agent（`agents/analyzer.md`）读两版 skill + 转写，找赢家强项 / 输家弱项 + 改进项，写 `analysis.json`。
5. **记胜负账**：更新 workspace 根 `history.json`（`current_best` 按 **test 分**择优、`grading_result` ∈ baseline/won/lost/tie，schema 见 `references/schemas.md §10`）。
6. **生成对比 HTML**：`generate_review.py "<ws>/iteration-<N+1>" --skill-name <名> --previous-workspace "<ws>/iteration-<N>" --static <out.html>` → 交人审。
7. **停机条件**：人满意 / 反馈全空 / 进展停滞（连续两轮 tie 或 lost）。

> 饱和信号：可解任务全过、无提升空间 = 饱和；若**基线（不带 skill）也过**，说明判据已被模型默认吸收 → 该 skill 可议退役。

覆盖 / 替换现役 skill 前，先出「报告 + 改动提案」交人确认，不直接覆盖——此步不可逆。
